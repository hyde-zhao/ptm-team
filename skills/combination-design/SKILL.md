## 目标

对设计计划中 PPDCS 特征为 **C-Combination** 的逻辑用例，执行五步设计过程，
因子表建模→Pairwise生成（PICT工具优先）→组合→LC→因子实际取值→输出物理用例。

## 理论基础

C-Combination 是 PPDCS 五特征之一：
> 被测功能涉及多个因子（参数/条件），每个因子有多种取值/状态，
> 全组合不可枚举，需要使用组合压缩策略。

**识别条件**：因子数 ≥ 3 且各因子取值数 ≥ 2，全组合超过合理用例数。

**关键区分**：
- D-Data vs C-Combination — 因子独立验证够？够 = Data，不够/需交互 = Combination
- P-Parameter vs C-Combination — 规则确定？确定 = Parameter，需探索 = Combination

**建模工具**：Pairwise（两两组合）/ 正交阵列（L 阵列）

## 适用范围

- 适用阶段：MFQ 的 design 阶段
- 输入：`.output/integration/design-plan.md`（PPDCS=C-Combination 的 LC）

## 五步用例设计过程

读取 `test-point-integrator` 输出的逻辑用例（含因子-取值表和动作路径），执行以下五步：

### 第一步：测试数据（因子-取值表补全）

| 告警级别 | 主要 | 参数 | 有效 | — |
| 告警级别 | 次要 | 参数 | 有效 | — |
| 告警级别 | 提示 | 参数 | 有效 | — |
| 告警类型 | 安全 | 参数 | 有效 | — |
| 告警类型 | 网络 | 参数 | 有效 | — |
| 告警类型 | 系统 | 参数 | 有效 | — |
| 告警类型 | 硬件 | 参数 | 有效 | — |
| 通知方式 | 邮件 | 参数 | 有效 | — |
| 通知方式 | 短信 | 参数 | 有效 | — |

### 第二步：Pairwise 生成（含 PICT 工具检测）

**PICT 工具检测**（优先使用自动化工具）：

```
工具检测顺序：
1. 检测 pict_wrapper.py 是否存在（路径：.output/scripts/pict_wrapper.py）
   → 若存在：生成 .pict 模型文件，调用 pict_wrapper.py 生成 Pairwise 组合表
   → 若不存在：执行手动 Pairwise 步骤

手动 Pairwise 规则（PICT 不可用时）：
  a. 列举所有因子对 (A,B), (A,C), ..., (N-1,N)
  b. 对每个因子对，确保所有取值组合至少出现1次
  c. 使用贪心覆盖策略：每新增一行，优先覆盖当前最少被覆盖的因子对取值
```

**PICT 模型文件格式**（适用 pict_wrapper.py 时）：

```
# 告警通知配置 - PICT 模型
告警级别: 紧急, 主要, 次要, 提示
告警类型: 安全, 网络, 系统, 硬件
通知方式: 邮件, 短信, Syslog, SNMP
频率限制: 无限制, 每分钟, 每小时, 每天
是否启用: 启用, 禁用

# 约束（过滤非法组合）
IF [告警级别] = "紧急" THEN [频率限制] <> "每天";
```

**C-Combination 的操作动作通常相同**（配置各因子取值，观测结果），通常为 1 条路径：

```
路径 P1（唯一路径）：
1. 进入告警通知配置页面
2. 按测试数据配置各因子（告警级别/类型/通知方式/频率/启用状态）
3. 保存配置
4. 触发对应告警事件
5. 观测通知是否发出及内容
```

若测试目标是"配置功能本身"而非"通知触发行为"，可拆分为 2 条路径，分别验证。

### 第三步：数据组合分析（Pairwise/正交为主）

**组合策略选择**：

| 条件 | 策略 |
|------|------|
| 因子数 ≤ 3 且相互独立 | 全组合 |
| 因子数 ≥ 4 且相互独立 | **Pairwise（两两覆盖）** |
| 因子间存在强约束 | 先过滤不合法组合，再对剩余组合应用 Pairwise |
| 因子取值数相同 | 可用正交阵列（L₁₆等） |

**本次选用：Pairwise**（5个因子，各有2~4个取值）

**组合约束分析**：

```
约束规则：
IF 告警级别=紧急 AND 频率限制=每天 → 该组合无效（紧急告警不允许每天限频）
IF 是否启用=禁用 → 预期结果=无通知（与告警级别/类型/通知方式无关）
```

### 第四步：因子实际取值分配

**C-Combination 的因子取值分配**：在 Pairwise 生成的组合框架上，为每个因子分配实际业务取值（而非第一步的抽象等价类标签）。

```
取值分配原则：
1. 有效等价类：选取典型代表值（如 IP 用 "<IP_ADDRESS>"，而非 "有效IP"）
2. 边界等价类：使用边界的精确值（如 "端口=65535" 而非 "最大端口"）
3. 无效等价类：选取最典型的无效值（如 "端口=0" 或 "端口=65536"）
4. 约束组合：确认约束过滤后的实际值仍满足约束（如 "紧急+频率≠每天"）
```

## 组合爆炸检测

- 全组合数 > 50 → 自动建议使用 Pairwise 压缩
- Pairwise 后 > 30 → 建议基于风险筛选
- 有约束条件 → 过滤不合法组合

## 优先级分配规则

| 组合类型 | 优先级 |
|---------|--------|
| 全有效因子的典型组合 | P1 |
| Pairwise 生成的覆盖组合 | P2 |
| 含约束边界的组合 | P2~P3 |
| 含无效因子的负面测试 | P3 |

## Gotchas

- 本 Skill 专注于 C-Combination（组合爆炸压缩），不处理 D-Data（独立数据验证）
- 无效等价类通常单独测试（一次只变一个无效值），不做无效值间组合
- 有约束关系的因子组合需要过滤不合法的组合
- 保留所有测试点到物理用例的追踪关系
