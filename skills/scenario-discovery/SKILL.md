## 目标

通过 MCP 知识库查询或 Web 搜索，获取目标特性（华为防火墙设备）的典型用户应用场景，
以交互方式与用户对齐场景列表，输出场景文档作为后续 M/F 分析的输入。

## 适用范围

- 适用阶段：MFQ 分析的 scenario 阶段
- 输入：特性名称 + 基本描述（来自 feature-parser 输出）
- 输出：`.output/scenarios/confirmed-scenarios.md`

## 前置条件

- [ ] feature-parser 已完成，`.output/feature-input/` 目录存在
- [ ] 特性名称和基本描述已确定

## 搜索策略

按以下优先级获取场景信息：

```
1. MCP 知识库查询（首选）
   ↓ 未命中
2. Web 搜索（回退）
   ↓ 补充
3. 用户提供的材料
```

### MCP 知识库查询

调用 `scripts/mcp_query_client.py` 查询知识库：

```bash
python scripts/mcp_query_client.py --query "<特性名称> 应用场景" --type scenario
```

若 MCP 未配置或无结果，自动回退到 Web 搜索。

### Web 搜索

使用平台内置搜索能力（如可用），搜索关键词：
- `华为防火墙 <特性名称> 应用场景`
- `华为 NGFW <特性名称> 部署方案`
- `Huawei firewall <特性英文名> use case`

### 用户材料补充

用户可能在 feature-parser 阶段提供了应用场景文档，从中提取场景信息。

## 执行流程

### 步骤 1：信息收集

1. 读取 `.output/feature-input/raw-requirements.md` 获取特性概况
2. 尝试 MCP 查询 → 无结果则 Web 搜索
3. 提取候选场景列表

### 步骤 2：场景结构化（三层结构）

将收集到的场景信息结构化为三层标准格式：

**层1：场景原理**（在场景确认阶段完成）

| 字段 | 说明 |
|------|------|
| 场景编号 | `SCN-<特性缩写>-NNN` |
| 场景名称 | 简明场景描述 |
| 场景分类 | 主场景 / 扩展场景 |
| 业务背景 | 为什么需要此场景，业务驱动因素（2~3句） |
| 核心目标 | 用户期望解决的问题或达到的目标 |
| 适用范围 | 设备型号/版本/配置前提下适用 |
| 不适用 | 明确排除的情况 |
| 关联模块 | 四/五级目录模块 |

**层2：使用路径（每个场景含1条主路径 + N条扩展路径）**（在场景确认阶段完成）

每条路径由有序节点构成，每个节点包含：

| 节点字段 | 说明 |
|---------|------|
| 节点编号 | N1, N2, N3... |
| 节点类型 | 用户输入 / 系统处理 / 用户操作 / 系统输出 |
| 操作描述 | 用户做什么 OR 系统处理什么 |
| 观察点 | 用户在该节点可观察到的界面/日志/状态变化 |
| 预期状态 | 该节点完成后的期望中间状态 |

路径类型：
- **主路径 P1**：正常流程（所有前置条件满足，操作正确）
- **扩展路径 P2+**：错误处理、边界行为、并发场景等

**层3：测试数据叠加**（在 M 分析完成后触发，Q3 default）

| 字段 | 说明 |
|------|------|
| 叠加节点 | 被叠加测试数据的路径节点编号（N1, N2...）|
| 数据维度 | 等价类 / 边界值 / 特殊值 |
| 测试数据集 | 具体数据取值列表（含典型值、边界值、异常值）|
| 关联TP-ID | 提供该数据维度的 M 分析测试点编号 |

> **触发时机**：层3 在 m-analysis 阶段输出 `test-points.md` 后叠加，不阻断场景确认流程。
> 若某场景无关联测试数据，层3 标注"M 分析未产出关联数据维度，层3 暂无叠加内容"。

### 步骤 3：用户交互确认

以表格或列表形式展示候选场景，使用 `ask_user` 工具发起结构化确认：

```
## 应用场景确认

特性：<特性名>

已发现 N 个典型场景（含层1+层2信息，层3 在 M 分析完成后自动叠加）：

| # | 场景名称 | 分类 | 主路径数 | 关联模块 |
|---|---------|------|---------|---------|
| 1 | ... | 主场景 | N条 | ... |
| 2 | ... | 扩展场景 | N条 | ... |
```

**ask_user 选项**：
1. ✅ 全部确认 — 场景列表完整，写入 confirmed-scenarios.md（层3 在 M 分析后自动叠加）
2. ✏️ 需要修改 — 请指出需要修改的场景编号及修改内容（名称/路径节点），修改后重新确认
3. ➕ 需要新增 — 请描述需要新增的场景（业务背景+关键路径），补充后重新确认
4. 🗑️ 需要删除 — 请指出需要删除的场景编号，删除后重新确认
5. ➕ 补充路径 — 请指出场景编号，补充缺失的扩展路径

### 步骤 4：输出持久化

将确认后的场景写入 `.output/scenarios/confirmed-scenarios.md`，使用三层结构：

```markdown
# <特性名> — 已确认应用场景

## 场景概览

| 编号 | 名称 | 分类 | 主路径数 | 关联模块 |
|------|------|------|---------|---------|
| SCN-XXX-001 | ... | 主场景 | N条 | ... |

---

## SCN-XXX-001: <场景名称>

**分类**：主场景

### 层1：场景原理

| 字段 | 内容 |
|------|------|
| **业务背景** | <为什么需要这个场景> |
| **核心目标** | <用户期望解决的问题> |
| **适用范围** | <适用设备/版本/前提> |
| **不适用** | <明确排除的情况> |
| **关联模块** | <四级 > 五级> |

### 层2：使用路径

#### 主路径 P1：<路径名称>

```
输入 → [N1] → [N2] → [N3] → 输出
```

| 节点 | 类型 | 操作描述 | 观察点 | 预期状态 |
|------|------|---------|--------|---------|
| N1 | 用户输入 | <用户输入什么> | <界面/回显/日志> | <期望状态> |
| N2 | 系统处理 | <系统处理什么> | <可观察的中间状态> | <期望状态> |
| N3 | 系统输出 | <最终输出> | <最终产物形式> | <终态> |

#### 扩展路径 P2：<路径名称>

| 节点 | 类型 | 操作描述 | 观察点 | 预期状态 |
|------|------|---------|--------|---------|
| N1 | 用户输入 | <异常输入> | <错误反馈> | <错误处理状态> |

### 层3：测试数据叠加（M 分析完成后填充）

| 叠加节点 | 数据维度 | 测试数据集 | 关联TP-ID |
|---------|---------|-----------|----------|
| N1 | 边界值 | <如：0台、1台、5台（上限）、6台（超限）> | TP-M-XXX-001 |
| N2 | 等价类 | <如：有效IP: 1.1.1.1；无效IP: 999.999.999.999> | TP-M-XXX-002 |

> 若 M 分析未产出与本场景关联的测试数据维度，标注：
> "M 分析未产出关联数据维度，层3 暂无叠加内容。"
```

## Gotchas

- Web 搜索结果可能包含过时信息（旧版本特性），需标注版本适用性
- 华为产品文档常使用特定术语（如"安全域"、"安全策略"），需保留原始术语
- 不同型号（TGFW vs NGFW）的场景可能有差异，需与用户确认适用范围
- 场景过多时（> 10 个），建议分主场景/扩展场景两类，先确认主场景
- 层3 叠加为追加写入，不修改已确认的层1/层2 内容

## 验收标准

- [ ] 每个场景包含三层结构（场景原理 / 使用路径 / 测试数据叠加）
- [ ] 层1+层2 在场景确认阶段完成（用户已确认）
- [ ] 层3 在 M 分析完成后追加填充（可为"暂无叠加内容"）
- [ ] 每个场景至少有 1 条主路径，路径节点含操作+观察点+预期状态
- [ ] 场景关联到四/五级目录模块
- [ ] `confirmed-scenarios.md` 已写入 `.output/scenarios/`
- [ ] `.output/doc/STATE.md` 更新
