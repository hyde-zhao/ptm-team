## 目标

当用户提供问题单（缺陷报告）时：
1. 分析问题单在当前用例框架下是否可以被覆盖
2. 不能覆盖的，定位在 MFQ 哪个环节出现了设计遗漏
3. 补充缺失的用例
4. 提供流程优化方法，防止同类遗漏再次发生

## 适用范围

- 适用阶段：MFQ 扩展分支（已有用例发现了问题单后）
- 输入：问题单列表 + 已有 `.output/` 全量数据
- 输出：盲区分析报告 + 补充用例 + 流程优化建议

## 前置条件

- [ ] 首次 MFQ 分析和用例设计已完成
- [ ] 用户提供了问题单（缺陷描述）

## 执行流程

### 步骤 1：问题单解析

从用户提供的问题单中提取关键信息：

| 字段 | 说明 |
|------|------|
| 问题编号 | 缺陷追踪系统编号 |
| 模块 | 问题所属模块 |
| 描述 | 缺陷详细描述 |
| 重现步骤 | 缺陷重现步骤 |
| 根因 | 根本原因（如有） |
| 严重程度 | 严重/一般/轻微 |

### 步骤 2：覆盖回溯

对每个问题单，在现有追踪链中反向查找：

```
问题描述 → 匹配 PC（物理用例）？
    ├── 找到 PC → 该问题在当前用例中可覆盖 → 标记"已覆盖但未检出"
    └── 未找到 PC → 继续反向查找
        → 匹配 LC（逻辑用例）？
            ├── 找到 LC 但无匹配 TD → 遗漏：测试数据不足
            └── 未找到 LC → 继续查找
                → 匹配 TP（测试点）？
                    ├── 找到 TP 但未转化为 LC → 遗漏：整合阶段丢失
                    └── 未找到 TP → 遗漏：MFQ 分析阶段缺失
                        → 判定是 M/F/Q 哪个阶段遗漏
```

### 步骤 3：遗漏环节定位

输出每个问题单的遗漏分析：

```markdown
## 遗漏分析报告

| 问题编号 | 覆盖状态 | 遗漏环节 | 遗漏原因 | 补充建议 |
|---------|---------|---------|---------|---------|
| BUG-001 | 已覆盖但未检出 | 测试数据 | TD 缺少边界值 | 补充边界数据 |
| BUG-002 | 未覆盖 | M 分析 | 功能点遗漏 | 新增 TP + LC + PC |
| BUG-003 | 未覆盖 | F 分析 | 耦合关系未发现 | 新增耦合 TP |
| BUG-004 | 未覆盖 | Q 分析 | 可靠性场景遗漏 | 新增质量 TP |
| BUG-005 | 未覆盖 | 整合 | TP 存在但未转 LC | 补充 LC |
```

### 遗漏环节分类

| 遗漏环节 | 说明 | 典型原因 |
|---------|------|---------|
| M 分析 | 功能点/测试点遗漏 | 需求理解不完整、功能拆分粒度不够 |
| F 分析 | 耦合关系未发现 | 矩阵不完整、场景耦合遗漏 |
| Q 分析 | 质量属性遗漏 | 相关维度未展开分析 |
| 整合 | 测试点未转化为逻辑用例 | 合并过程中丢失 |
| 设计 | 设计方法未覆盖路径/状态/组合 | 路径枚举不完整 |
| 测试数据 | 数据不足以触发缺陷 | 等价类划分不充分 |

### 步骤 4：用例补充

对未覆盖的问题单：

1. **回到遗漏环节**：从遗漏的阶段开始补充
2. **增量分析**：仅补充遗漏的部分
3. **用例设计**：按推荐方法设计补充用例
4. **不修改现有用例**：仅新增，不改变不受影响的用例

### 步骤 5：流程优化建议

分析遗漏的系统性原因，提供工作流优化建议：

```markdown
## 流程优化建议

### 问题模式

| 遗漏环节 | 频次 | 模式分析 |
|---------|------|---------|
| M 分析 | 3次 | 异常场景功能点覆盖不足 |
| F 分析 | 2次 | 场景耦合推理覆盖不全 |

### 优化措施

1. **M 分析优化**：增加异常场景分析检查项
   - 新增检查维度：异常输入、并发操作、资源耗尽
   - 影响：m-analyzer Skill 的分析维度增加 3 项

2. **F 分析优化**：增加耦合发现规则
   - 新增规则：跨模块事件传播链路分析
   - 影响：f-analyzer Skill 增加推理规则
```

## 不可变保护

与 `change-impact-analyzer` 相同，不修改不受影响的用例。

## Gotchas

- "已覆盖但未检出" 的情况说明用例设计没问题，但执行可能有问题
- 问题单的描述可能模糊，需要与用户确认具体的模块归属
- 流程优化建议应该是具体可操作的，不要泛泛而谈
- 补充用例时优先考虑和同类问题相关的场景

## 验收标准

- [ ] 每个问题单的覆盖状态和遗漏环节已识别
- [ ] 未覆盖的问题单有具体的补充用例
- [ ] 不受影响的用例未被修改
- [ ] 流程优化建议具体可操作
- [ ] 分析报告和补充用例已写入
