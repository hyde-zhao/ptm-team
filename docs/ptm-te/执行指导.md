> 定位：指导性文档，覆盖从环境准备到报告解读的完整链路。重操作、讲关键原理，控制篇幅。

## 1. 概述

**ptm-te 是测试执行编排器**：消费 ptm-tde 产出的物理用例（PC），在真实设备上按八步流程执行原子操作并回写结果。它本身不做测试设计，只做"翻译 + 执行 + 判定 + 清理"。

### 完整链路

```
环境准备 ──► 设备准备 ──► 用例转换 ──► 用例执行 ──► 报告解读
  ptm-atomic   devices.yaml   16列表          八步流程        runs/<run-id>/
  uv/Python    .env/可达性     ->case_steps    op_mapper       result.json
  ptm-te agent  快照/session   参数适配        清理回滚        report.md
```

每阶段产物流向下一阶段：环境就绪才能探测设备，设备可达才能转换用例参数，用例转成 `case_steps` 才能执行，执行产出 `runs/<run-id>/` 供解读。

### 前置依赖

| 依赖 | 用途 | 安装方式 |
|------|------|---------|
| ptm-atomic CLI | 真实执行 op（auth/policy-route/object 等） | `uv tool install` |
| uv | 管理 Python 环境（隔离 paramiko/telnetlib） | 系统包管理 |
| Python ≥3.9,<3.13 | telnetlib 兼容（3.13 移除） | uv 管理 |
| ptm-te agent + 3 skill | 编排器 + device-management/connection/policy-route-execution | `ptm-team install claude --agent ptm-te` |
| 设备网络可达 | Web 443 / SSH 22 / Telnet | TCP 探测确认 |

### 版本与已知偏差

本文基于 **op_mapper 15 op_id 覆盖**（CR-028 扩展后，非 README.md 旧的 8 op_id）。ptm-atomic 0.1.0 安装版实测：`policy-route update --help` 已注册 `--id`（required），update 操作可用（原"未注册 --id"已修复，详见 §7 工具类陷阱）。

#### 2.1.1 ptm-atomic 依赖锁定（P0-2，CR-032）

ptm-te 执行依赖的 ptm-atomic 版本必须锁定，避免重装后行为漂移（如 source_network 全 any 缺陷、trex 子命令缺失）。当前锁定值同步记录于 ptm-te 仓库 `.ptm-team-manifest.json` 顶层 `ptm_atomic_dep`：

| 项 | 值 |
|----|----|
| 仓库 | ptm-atomic（`/home/hyde/projects/ptm-atomic`） |
| commit | `952a6252cb6d649578a05572bcd08161323157a3`（短 `952a625`） |
| 分支 | main |
| 版本 | 0.1.0 |

重装时按此 commit 检出再 `uv tool install`，确保复现同一版本。manifest 顶层 `ptm_atomic_dep` 由 install.py `load_manifest`/`save_manifest` 保留（仅更新 `installs` 数组），重装 ptm-te agent 不会覆盖该字段。

#### 2.1.2 重装 ptm-atomic 前置检查（P0-1，CR-032）

重装 ptm-atomic 前后必须执行下列检查，避免装到不含 trex 或含已知缺陷的版本导致用例回归。三项验证全过才进入用例执行。


### 2.2 Python 运行环境（telnetlib 陷阱）

**原理**：`collect_sysinfo.py` 依赖 `paramiko`（SSH）+ `telnetlib`（标准库，**Python 3.13 已移除**）+ `PyYAML`。系统 Python 通常无 paramiko、无 pip，直接 `python3` 跑必失败。uv 默认拉取最新 Python（3.13+），缺 telnetlib。

**方案**：用 uv 显式锁定 Python 3.12 + 临时依赖，不污染系统：

```shell
uv run --python 3.12 --with paramiko --with pyyaml \
  python <script>.py <args>
```

> 系统无 `pip3`/`ensurepip` 时，uv 是唯一可行方案。校验：`uv python list | grep 3.12`。

## 3. 设备准备

### 3.1 添加设备到 devices.yaml

**原理**：凭据用 `${ENV_VAR}` 占位，实际值放 `.env`，两者都 gitignore，避免明文凭据入库。运行时由 `collect_sysinfo.py` 从环境变量解析。

```yaml
device_groups:
  - name: nxp290-171
    description: "防火墙设备 NXP290-171"
    firewall:
      host: <IP_ADDRESS>
      role: firewall
      device_type: DAS-TGFW-290        # 查 reference/device-reference.md 确定
      web:    { host: <IP_ADDRESS>, port: 443, user: admin, password: ${FW_WEB_PASSWORD} }
      telnet: { host: <IP_ADDRESS>, port: 10004, user: root, password: ${FW_TELNET_PASSWORD} }
      ssh:    { host: <IP_ADDRESS>, port: 22, user: root, password: ${FW_SSH_PASSWORD}, enabled: true }
    tags: [firewall, nxp1043]
```

`device_type` 按 [device-reference.md](../../../ptm-te/.claude/skills/device-management/reference/device-reference.md) 查表确定（硬件系列 -> TGFW 型号）。

### 3.2 配置 .env

```shell
set -a; source .env; set +a    # 每次执行前加载到环境
```

```env
FW_WEB_PASSWORD=...
FW_SSH_PASSWORD=...
FW_TELNET_PASSWORD=...
```

### 3.3 设备可达性探测

**原理**：部分设备禁 ICMP（ping 不通但端口可达），故用 TCP 端口探测而非 ping。

```shell
timeout 3 bash -c 'cat </dev/null >/dev/tcp/<IP_ADDRESS>/443' && echo "Web可达"
timeout 3 bash -c 'cat </dev/null >/dev/tcp/<IP_ADDRESS>/22'  && echo "SSH可达"
```

### 3.4 采集系统快照 before

```shell
RUN_ID="run-$(date +%Y%m%d-%H%M%S)"
mkdir -p runs/$RUN_ID/snapshot-before
uv run --python 3.12 --with paramiko --with pyyaml python \
  .claude/skills/device-connection/scripts/collect_sysinfo.py \
  nxp290-171 --phase before --run-id $RUN_ID
```

输出 `runs/<run-id>/snapshot-before/<device>.json`，含 cpu/memory/disk/processes 4 维度（SSH 首选，失败回退 Telnet）。

### 3.5 设备参数探查

**原理**：用例参数常缺 `in_interface`，且 `--source-network` 期望地址对象名而非 CIDR。需先探查设备的路由模式接口名。

```shell
# SSH 查接口（找 UP + 有 IP 的路由口）
ip -4 addr show | grep -B1 "scope global"
# nxp290-171 实测：GE0_1（<IP_ADDRESS>/22, UP）是唯一路由模式接口
```

地址对象命名规则：**仅字母、数字、连字符**（下划线被拒，如 `OBJ_SRC` -> `OBJ-SRC`）。

### 3.6 登录建立 session

**原理**：ptm-atomic 拒绝 session 文件写在仓库目录内（`RUNNER_SESSION_INVALID`），必须用 `~/.local/state/ptm-atomic/` 下路径；省略 `--execute` 则只 dry-run，不建真实 session。

```shell
SESSION=~/.local/state/ptm-atomic/ngfw/session-$RUN_ID.json
ptm-atomic run --base-url https://<IP_ADDRESS> --session-file $SESSION \
  --execute auth login --username admin --password-env FW_WEB_PASSWORD
# 预期 status: success, state_status: valid
```

`--password-env` 传环境变量名（非明文密码），`--verify-tls` 默认关闭（适配自签证书）。

## 4. 用例转换（16 列表 -> case_steps）

### 4.1 PC 消费契约（CR-019）

**原理**：ptm-te **不消费 16 列汇总表**（仅人工 fallback），只消费结构化 `case_steps` 的 4 个字段：

| 字段 | 路径 | 用途 |
|------|------|------|
| `step_name` | `case_steps[].step_name` | 测试动作意图 |
| `op_id` | `case_steps[].atomic_op.op_id` | 原子操作标识 |
| `args` | `case_steps[].atomic_op.args` | 操作参数 |
| `expected_result` | `case_steps[].expected_result` | 预期结果（判定比对） |

### 4.2 16 列汇总表速读

ptm-tde 产出的 16 列表（`ppdcs/delivery/<特性>特性测试用例.md`）每行一个 PC，关键列：

- **用例编号**（如 `PC-M1-01-01`）、**用例名称**、**用例级别**（P0/P1/P2）
- **预置条件**：前置数据（如"预置策略路由 ID=pr-001"）
- **测试步骤**：用 `<br>` 分隔多步，每步标注 `原子操作：fw_xxx`
- **预期结果**：判定依据
- **关键词**：含 op_id，可快速判断是否含流量操作（`tg_*`）

筛选可执行用例：排除含 `tg_start_traffic_stream`/`tg_capture_traffic_result`/打流的行（需流量仪，单设备无法执行）。

### 4.3 转换模板

```yaml
case_steps:
  - step_id: STEP-001
    step_name: 管理员登录 DUT
    target: DUT
    atomic_op:
      op_id: fw_login_web_management
      args: { username: admin, password_env: FW_WEB_PASSWORD }
    expected_result: 登录成功，session 建立
  - step_id: STEP-002
    step_name: 创建策略路由（主操作）
    target: DUT
    atomic_op:
      op_id: fw_config_policy_route
      args:
        source_network: OBJ-SRC-192      # 地址对象名，非 CIDR
        next_hop_ip: 3.3.7.1
        in_interface: GE0_1               # 路由模式接口
        type: ipv4
    expected_result: 创建成功，返回策略 ID
    depends_on: STEP-001                  # 跨步骤依赖
```

### 4.4 参数适配规则

**原理**：用例原文参数常是描述性（"源IP=<IP_ADDRESS>/24"），需对齐 ptm-atomic op yaml 的 `inputs.params` 要求（CR-025 后 args 命名与 params 一致，三层退化为两层）。

| 用例原文 | 适配为 | 原因 |
|---------|--------|------|
| 源IP `<IP_ADDRESS>/24` | `source_network: OBJ-SRC-192` | `--source-network` 要地址对象名；CIDR 含 `/` 触发 `eParamIllegal` |
| 入接口缺失 | `in_interface: GE0_1` | config 必填，需路由模式接口（§3.5 探查） |
| `src_addr`（旧命名） | `source_network` | CR-025 对齐 op yaml params |
| id（update/delete） | `id: ${STEP-001.id}` | 由 op_mapper 按被引 op 的 `id_source` 声明自动解析（response/args/query/placeholder 4 模式）；替代旧的 `<policy_route_id-from-config:STEP-N>` 约定 |

### 4.5 op_id 选择与替换

**原理**：op_mapper 当前覆盖 15 op_id（5 族，见 [op-coverage-matrix.md](op-coverage-matrix.md)）。用例 op_id 必须在覆盖内才能走编排；不在的需替换为 ptm-atomic 实际存在的 op_id。

常见替换：

| 用例原文 op_id | 替换为 | 原因 |
|---------------|--------|------|
| `fw_verify_operation_log`（gap CAP-003） | `fw_capture_operation_log` | ptm-atomic 实际命名是 capture（抓取日志）非 verify |
| `fw_check_login_state` | `fw_login_web_management` | 前者是契约 guard，无实现；映射为 login |

### 4.6 转换后校验

```shell
python3 -c "
import sys; sys.path.insert(0,'.claude/skills/policy-route-execution/scripts')
import op_mapper
# 对每个 step：op_id in OP_ID_TO_SUBCOMMAND？args key in ARGS_TO_FLAGS[op_id]？
"
```

> 完整校验脚本见 [附录 E](#附录e-用例转换校验脚本片段)。

## 5. 用例执行

### 5.1 dry-run vs --execute

**原理**：默认 `--dry-run`（只打印 request_plan，不改设备）；`--execute` 是**全局 flag**，真实发请求。写操作（config/update/delete）必须 `--execute`，且属 DQ-01 单次授权门控——执行前需向用户确认。

**DQ-01 授权粒度（P2，CR-032）**：一次 `--execute` 授权的覆盖范围如下，避免子集重跑时授权语义模糊：

| 维度 | 规则 |
|------|------|
| 覆盖范围 | 本轮 exec_v4 的全部用例（整轮 17 例或 `--cases` 子集） |
| 子集重跑 | 复用本轮授权，无需逐用例重新确认（exec_v4_subset.py 当前行为；批次 B 后由 `exec_v4.py --cases` 承载） |
| 跨轮延续 | 不延续--新一轮 exec_v4 启动需重新获得 `--execute` 授权 |
| 记录 | 每条 `result.json` 的 `runtime_authorization` 记录授权主体（DQ-01）/范围/时间；批次 B 将补 `scope_note` 字段与启动授权声明 |
| 边界 | 设计通过（CP2/CP3）不等于运行授权；真实设备写操作必须独立授权 |

```shell
# dry-run（不授权，验证参数路由）
ptm-atomic run --base-url $URL --session-file $S policy-route config ... 
# --execute（授权后真实写）
ptm-atomic run --base-url $URL --session-file $S --execute policy-route config ...
```

### 5.2 八步编排流程

| 步 | 动作 | 输入 | 产物 |
|----|------|------|------|
| [1] | 用例解析 | `cases/upload/*.md` | `parse-result.json` |
| [2] | 设备准备 | devices.yaml + .env | `snapshot-before/` |
| [3] | login 一次 | Web 凭据 | session.json（复用） |
| [4] | 逐条 op 执行 | op_mapper 翻译 | envelope（逐条） |
| [5] | 结果判定 | envelope vs expected | check_passed |
| [6] | 执行日志 | 逐条结果 | `exec-log.jsonl` |
| [7] | 用例清理 | inverse_op | cleanup_summary |
| [8] | 快照 after + 回写 | - | `snapshot-after/` + `result.json` + `report.md` |

### 5.3 session 复用与失效重连

**原理**（login-once-reuse-session）：[3] login 一次持久化 session，后续 config/verify/update/delete 复用同一 session。遇 `STATE_INVALID`（session 过期）自动重新 login + 重试当前 op（最多 1 次）。login 失败则终止用例（`AUTH_FAILED`）。

### 5.4 逐条 op 执行命令

```shell
B="ptm-atomic run --base-url https://<IP_ADDRESS> --session-file $SESSION --execute"
$B object config --object-name OBJ-SRC-192 --ipaddr <IP_ADDRESS> --mask 24   # 创建地址对象
$B policy-route config --source-network OBJ-SRC-192 --next-hop-ip 3.3.7.1 --in-interface GE0_1 --policy-route-type ipv4
$B policy-route verify --policy-route-type ipv4 --page 1 --size 10           # 查询
$B policy-route delete --id 1 --policy-route-type ipv4                       # 删除
```

### 5.5 步骤间 id 传递（${STEP-N.id}）

**原理**：后续步骤引用前序步骤创建的资源的 id，不再需要 LLM 手动从 envelope 提取。在 PC 的 `args` 中用 `${STEP-N.id}` 占位符，op_mapper 的 `resolve_step_refs` 会按被引步骤 op 的 `rollback_strategy.id_source` 声明自动解析（经 `ptm-atomic show` 读取），支持 4 种模式：

| id_source   | 解析方式                                              | 示例 op                          |
|-------------|--------------------------------------------------------|----------------------------------|
| `response`  | 取被引 step envelope.data[id_field]                    | fw_config_policy_route           |
| `args`      | 取被引 step args[id_field]（id 即 name）               | fw_config_object                 |
| `query`     | 执行 verify GET 查询，按 name 匹配取 id                | fw_config_acl_policy（待 op 覆盖扩展） |
| `placeholder`| id 取固定占位（如 "1"），真正靠其他字段定位             | fw_update_acl_policy_group      |

**步骤间引用示例**：
```yaml
  - step_id: STEP-001
    atomic_op:
      op_id: fw_config_policy_route   # id_source=response, id_field=policy_route_id
      args: {source_network: OBJ_SRC, in_interface: GE0_1, type: ipv4}
  - step_id: STEP-002
    depends_on: STEP-001
    atomic_op:
      op_id: fw_update_policy_route
      args:
        id: ${STEP-001.id}            # 自动解析为 STEP-001 envelope.data.policy_route_id
        type: ipv4
        weight: 200
```

执行 op_mapper 时传入 `--step-refs-dir runs/<run-id>/step-refs/ --step-id STEP-002`，op_mapper 会自动：读取 `step-refs/STEP-001.json` → 调 `resolve_id` 按 fw_config_policy_route 的 id_source=response 声明取 policy_route_id → 替换 `args.id` 为实际值 → 构建命令执行。

若不用占位符，LLM 仍可手动从 envelope 提取 id 传入（向后兼容），但推荐占位符写法以确保声明驱动解析。

### 5.6 用例清理

**原理**：`handle_rollback` 按 op 的 `rollback_strategy` 策略清理，经 `ptm-atomic show` 读声明按 `id_source` 模式自动解析 id：

| id_source   | 回滚 id 来源                                  | 示例                              |
|-------------|-----------------------------------------------|-----------------------------------|
| `response`  | 前向 POST envelope.data[id_field]             | fw_config_policy_route → delete   |
| `args`      | 前向 args[id_field]（id 即 name）             | fw_config_object → delete*        |
| `query`     | 执行 verify GET 查询按 name 匹配              | fw_config_acl_policy → delete*    |
| `placeholder`| 固定占位 + 字段互换（如 rename-back）         | fw_update_acl_policy_group*       |

*标记的 op 的逆操作（fw_delete_object、fw_delete_acl_policy 等）在 ptm-atomic 侧已实现，但 ptm-te 的 op 覆盖（OP_ID_TO_SUBCOMMAND 等）尚未扩展，当前仍按 `type=none` 跳过回滚。扩展后可激活声明驱动回滚，不再需要手动 curl 清理。

object 删除时若被策略路由引用会报 `eBeingReferenced`，**必须先删策略路由再删对象**。

```shell
# 1. 先删策略路由（解除引用）
$B policy-route delete --id $PR_ID --policy-route-type ipv4
# 2. 再删地址对象（op 覆盖扩展后由 handle_rollback 接管，无需手动 curl）
TOKEN=$(python3 -c "import json;print(json.load(open('$SESSION'))['session_value'])")
curl -sk --noproxy '*' -X DELETE -H "Authorization: Bearer $TOKEN" \
  "https://<IP_ADDRESS>/api/v1/ippool?id=OBJ-SRC-192"
```

`--noproxy '*'` 跳过 WSL2 代理（设备 IP 走代理会失败）。`Authorization: Bearer <jwt>` 用 session 文件的 `session_value`。

### 5.7 异常降级

| error_type | 触发 | 处理 |
|-----------|------|------|
| `OP_NOT_FOUND` | op_id 未在 op_mapper | 阻塞，反馈 ptm-tae 补工具 |
| `EXEC_FAILED` | CLI 返回错误/异常 | 记录，继续或终止（如 ptm-atomic 执行异常） |
| `ENV_NOT_READY` | devices.yaml 缺失/明文凭据/设备不可达 | 终止该设备，降级 dry-run-only |
| `STATE_INVALID` | session 过期 | 自动重连 + 重试 1 次 |
| `AUTH_FAILED` | login 失败 | 终止用例 |

### 5.8 多用例顺序执行

**策略**：CRUD 闭环 + 前序产物复用。建议顺序：**只读验证（确认设备干净）-> create -> 重复create -> update -> delete**。前序用例创建的策略路由可供后续用例作前置，减少重复 config/delete。每个用例仍需自包含清理（[7] 回滚），保证设备可恢复。

### 6.2 exec-log.jsonl

每行一条 JSON，记录单步执行：

关键字段：`status`（success/error）、`error_type`（NONE/EXEC_FAILED 等）、`api_status_code`（200/400）、`note`（id/异常）。

### 6.3 result.json

```json
{
  "summary": {"total":5,"pass":4,"fail":1,"pass_ids":[...],"fail_ids":["PC-M1-01-02"]},

  "cleanup": "所有用例清理完成，count=0",
  "key_findings": ["config 非幂等（重复创建建新条目）", ...]
}
```

> 注：update --id 已于 ptm-atomic 0.1.0 修复可用，PC-M1-01-07 可正常执行（原"update bug"已消除）。

### 6.4 report.md

人类可读：执行结果总览表 + 逐步骤结果表 + FAIL 根因分析（含 evidence）+ 清理记录 + 关键发现。优先看 `summary` 定位 PASS/FAIL，再读 FAIL 用例的 `reason`/`evidence`。

### 6.5 PASS/FAIL 判定标准

**原理**（envelope 契约）：每条 op 执行封装为 envelope，判定规则为 `status=success` + `error_type=NONE` + Check 点匹配 expected_result -> PASS；否则 FAIL。

- op 执行层面：`status=success && error_type=NONE` 表示该步 op 成功
- 用例层面：所有 step 的 Check 点（expected_result）匹配 -> 用例 PASS
- **注意**：op 成功 ≠ 用例 PASS。如 PC-M1-01-02 重复创建，op 返回 success（创建第二条），但用例预期"应失败"，Check 点不匹配 -> 用例 FAIL

### 6.6 常见 FAIL 排查

| FAIL 现象 | 根因 | 排查 |
|-----------|------|------|
| update 报缺 `--id` | 未从 config 响应提取 `policy_route_id` | config 执行后从 `envelope.data.policy_route_id` 取 id 传入 update `--id`（原"ptm-atomic 未注册 --id"已于 0.1.0 修复） |
| 重复创建返回 success 而非失败 | config 非幂等，重复创建建新条目 | 确认是否产品正常行为；修正用例预期 |
| `eParamIllegal` 参数校验失败 | source_network 用 CIDR / 对象名含下划线 | 改地址对象名 + 连字符命名 |
| `eObjParse` 被引用对象不存在 | source_network 引用的地址对象未创建 | 先 `object config` 创建对象 |
| `eBeingReferenced` 对象删不掉 | 策略路由仍引用该对象 | 先删策略路由再删对象 |

## 7. 已知陷阱速查（Gotcha）

### 环境类

| 现象 | 原因 | 规避 |
|------|------|------|
| `import telnetlib` ModuleNotFoundError | uv 默认 Python 3.13，telnetlib 已移除 | `uv run --python 3.12` |
| `No module named 'paramiko'` | 系统 Python 无 paramiko/pip | `uv run --with paramiko --with pyyaml` |
| `pip3: command not found` | 系统 Python 无 pip/ensurepip | 用 uv，不用 pip |

### 设备类

| 现象 | 原因 | 规避 |
|------|------|------|
| `RUNNER_SESSION_INVALID` session 写仓库内 | ptm-atomic 禁止 session 写仓库目录 | 用 `~/.local/state/ptm-atomic/` 下路径 |
| curl 设备失败 | WSL2 代理 https_proxy 拦截 | `--noproxy '*'` |
| `--in-interface` 被拒 `ePolicyRouteInIfModeError` | 接口非路由模式 | SSH 查 `ip addr`，用 UP + 有 IP 的路由口 |
| 接口名非 eth0 | NGFW 用 `GE0_N` 命名 | 探查设备实际接口名 |

### 工具类

| 现象 | 原因 | 规避 |
|------|------|------|
| ~~update 抛 AttributeError~~ | ~~`policy-route update` 未注册 `--id`~~ 已于 ptm-atomic 0.1.0 修复（`--help` 已暴露 `--id`） | 无需规避；update 正常用 config 返回的 `policy_route_id` 作 `--id` |
| 地址对象无法用 ptm-atomic 删除 | object 族只有 config，无 delete | curl `DELETE /api/v1/ippool?id=<name>` |
| 重复创建不拒绝 | config 非幂等 | 确认产品行为，必要时修正用例预期 |

### 用例类

| 现象 | 原因 | 规避 |
|------|------|------|
| `fw_verify_operation_log` 执行阻塞 | op_mapper 未覆盖（ptm-tde 用 verify，ptm-atomic 是 capture） | 替换为 `fw_capture_operation_log` |
| 参数描述性无法执行 | 16 列表原文是"新增一条策略路由"无具体值 | 转换时补全 in_interface/source_network 等 |
| `fw_check_login_state` 无映射 | 契约 guard，无实现 | 映射为 `fw_login_web_management` |
