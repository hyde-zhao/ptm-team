## 一、项目概览

### 1.1 使命

PTM Team（Product Test Method Team）是一个由 **6 个 AI Agent 角色** 组成的防火墙测试团队。目标是将测试团队从「AI 辅助」提升到「AI 原生」——AI 不仅是工具使用者，更是测试设计、执行、分析的主体。

### 1.2 核心设计原则

| 原则 | 含义 | 决策影响 |
|------|------|----------|
| **渐进落地** | Skill → 工作流 → Agent，单点跑通再组合 | 不做大跃进式全量开发 |
| **分层解耦** | 规划层 / 执行层 / 感知层独立演进 | 每一层可独立迭代 |
| **共建共用** | 公共 Skill 和工具由 ptm-tae 统一维护，所有角色按需调用 | 避免各位工程师重复造轮子 |
| **AI 原生** | 框架同时服务 AI 自治和手工测试辅助 | 不可替代手工时也能提供价值 |

### 1.3 三层架构

```
规划层（Planning）
  通用大模型 + Agent → 读取测试用例 → 规划测试步骤
  调用执行层工具执行测试行为，调用感知层工具 Check 执行结果

执行层（Execution）
  单个可执行工具 → TGFW 配置 / 组网切换 / 流量生成 / 设备管理

感知层（Perception）
  多模态感知 → 文本（日志、命令行）/ 图像（截图）/ 表格（性能数据）
  对照用例 Check 点给出 Pass/Fail 判定和失败原因摘要
```

### 1.4 四步演进愿景

```
Step 1 ──────────► Step 2 ──────────► Step 3 ──────────► Step 4
Copilot 准备      Copilot 人工主导   Copilot Agent 主导  Autopilot
工具开发、知识库   人工主导 Agent 按  Agent 主导流程，    全自动运行，
建设、Skill 建设   指定流程完成任务   人工辅助 Check      人工仅处理异常
```

**最终目标**：每位工程师拥有自己的数字员工——公共 Skill 和工具作为共享基础设施，个人元提示词和元工作流作为私有扩展。

#### PPDCS 五特征

| 特征 | 含义 | 推荐设计技术 |
|------|------|------------|
| **P-Process** | 多步骤有序流程，不可回退 | 流程图/活动图 |
| **P-Parameter** | 参数间有确定性业务规则 | 判定表/因果图 |
| **D-Data** | 数据有取值范围，各项独立 | 等价类+边界值 |
| **C-Combination** | 多因子多状态，需组合压缩 | Pairwise/正交阵列 |
| **S-State** | 对象多状态可双向迁移 | 状态图/转换表 |

#### CAE 三元组（核心数据模型）

```
TP-M-001:
  C（Condition）: 防火墙已启动，日志服务器处于"已配置"状态，IP=<IP_ADDRESS>
  A（Action）:    启用日志服务器（点击"启用"按钮）
  E（Effect）:    日志服务器状态变为"已启用"，系统日志记录操作记录
```

**追踪链**：`SR → TP(C/A/E) → LC（逻辑用例）→ 组合方案 → PC（物理用例，16 列）`

| 专属 Skill | 功能 |
|-----------|------|
| `testcase-design-skill` | 读取需求 → 生成测试点和用例框架 → 校验覆盖度 |
| `feature-analysis-skill` | 分析特性文档 → 提取关键参数和边界 → 生成测试矩阵 |
| `adversarial-test-skill` | 对抗性场景自动生成（边界攻击、配置冲突、并发压力） |

### 4.2 Step 1 — Copilot 准备阶段（当前阶段）

**目标**：搭建三个核心 Agent 及其底层工具能力，完成从 0 到 1 的基础设施建设。

**交付 Agent**：`ptm-tde`（测试设计）、`ptm-te`（测试执行）、`ptm-tae`（自动化/工具开发）。三个 Agent 形成最小闭环——tde 设计用例 → te 执行测试 → tae 提供工具支撑。

#### 4.2.1 原子能力

将测试操作抽象为可被 Agent 调用的原子能力，按设备类型和协议分层建设。

| 能力分类 | 内容 | 当前状态 | 交付要求 |
|---------|------|:---:|---------|

| **数通仪表打流** | BPS/IXIA 等仪表流量生成（吞吐、时延、丢包） | 🔄 进行中 | 支持 L2-L4 层基础打流模板、结果数据自动采集与结构化输出 |
| **应用层流量** | HTTP/HTTPS、FTP、Email、DNS 等应用协议流量模拟 | ⬜ 待开发 | 支持自定义 Header/Body/认证，覆盖应用层策略验证场景 |

#### 4.2.2 因子库

将测试经验沉淀为结构化因子数据，支撑 ptm-tde 的用例设计和 ptm-te 的测试执行。

| 因子库内容 | 说明 | 当前状态 | 交付要求 |
|-----------|------|:---:|---------|
| **策略路由因子** | PBR 相关的等价类、边界值、组合约束 | 🔄 部分完成 | 覆盖全部 PBR 配置参数，含正向/反向/异常因子 |
| **安全策略因子** | 源/目的地址、端口、协议、动作等组合 | ⬜ 待开发 | 支持 Pairwise 组合生成 |
| **NAT 因子** | SNAT/DNAT 的地址池、端口映射、ALG 等 | ⬜ 待开发 | 覆盖全部 NAT 类型和边界条件 |

> 因子库采用统一的 YAML schema，由 ptm-tde 设计格式，ptm-tae 实现解析工具，ptm-te 在执行中积累数据。

#### 4.2.3 Topo 管理

Topo 管理的核心思路：**物理设备资源池化 → 逻辑 Topo 描述需求 → 映射空闲资源 → 交换机隔离出并行环境**。

**两层 Topo 模型**：

```
逻辑 Topo                          物理 Topo
（描述需要什么）                    （实际用什么）
┌─────────────────┐          ┌─────────────────┐

| 层面 | 能力 | 说明 | 当前状态 | 交付要求 |
|------|------|------|:---:|---------|
| **逻辑 Topo** | 组网描述规范 | Mermaid + YAML 描述：需要哪些设备（DUT/TG/SW/Mock）、链路类型（GE/TE/电口/光口）、接口数量，不绑定具体物理设备 | 🔄 进行中 | 规范 v1.0 定稿，覆盖全部链路类型 |
| | 组网图集合 | 覆盖全部测试场景的标准化逻辑 Topo 模板（node2 ~ node8） | 🔄 进行中 | ≥ 15 个模板，含 VPN/NAT/HA/PPPoE 场景 |
| **物理 Topo** | 大网建设 | 所有物理设备全连接：防火墙 + 交换机 + 仪表 + Mock 服务器全部接入，构成统一资源池 | 🔄 进行中 | 物理设备全量接入，链路连通性验证通过 |
| | 设备使用状态管理 | 追踪每台物理设备的占用/空闲/故障状态，设备属性台账（型号、版本、端口类型） | ⬜ 待开发 | 实时状态查询、资源预约、冲突检测 |
| **Topo 映射** | 逻辑→物理映射 | 根据逻辑 Topo 的设备/链路需求，从空闲物理资源中自动匹配，生成可用的物理 Topo | ⬜ 待开发 | 自动匹配 + 人工确认，输出映射表 |
| | 组网校验 | 校验映射后的物理 Topo 是否符合逻辑 Topo 的约束（设备类型匹配、链路类型一致、接口数量足够） | ⬜ 待开发 | 校验脚本退出码 = 0 |
| **交换机隔离** | VLAN 切换 | 通过配置交换机 VLAN，将大网切分为多个独立二层域，每个域对应一套逻辑 Topo | ⬜ 待开发 | H3C/Huawei/锐捷三种交换机统一接口 |
| | VRF/路由隔离 | 通过 VRF 实例隔离三层路由，保证多套环境的路由表相互独立 | ⬜ 待开发 | 支持批量 VRF 创建/删除/切换 |
| | 并行环境管理 | 多套逻辑 Topo 同时映射运行，互不干扰；一键切换，配置备份与快速恢复 | ⬜ 待开发 | 同时运行 ≥ 3 套隔离环境 |

> **关键价值**：通过交换机配置隔离，一套物理设备可以同时支撑多套并行的测试环境，大幅提升硬件资源利用率。逻辑 Topo 一次定义、反复映射，不必每次手工搭建环境。

### 4.3 Step 2 — Copilot 人工主导

**目标**：人工主导，三个 Agent 各司其职，打通「需求分析 → 用例设计 → 任务执行 → 自动化翻译 → 回归运行」全链路。

**阶段特征**：人工掌握流程控制权——工程师使用 Agent 完成各自的工作，Agent 提供智能辅助但不自主决策。ptm-tde 辅助设计、ptm-te 辅助执行、ptm-tae 将执行过的用例翻译为自动化脚本。

**核心流程**：

│  ② ptm-tde 执行 12 步分析流程（场景发现 → M/F/Q 分析）│
│  ③ 人工在 5 个检查点确认关键产物                       │
│  ④ 输出：测试方案 + 测试用例（16 列物理用例）          │

#### 4.3.1 ptm-tde 需求分析与用例设计

| 步骤 | 活动 | 输入 | 输出 | 人工职责 |
|:---:|------|------|------|---------|
| 1 | 需求文档解析 | Markdown/Word/Excel/PDF 需求文档 | 结构化需求（SR 编号/模块/描述） | 确认需求版本和范围 |

| 3 | M 分析（单功能拆分） | 确认后的场景 | PPDCS 标注 + CAE 测试点 | 检查点②：目录结构确认 |
| 4 | F 分析（耦合分析） | M 分析产物 + 耦合矩阵 Excel | 耦合测试点（CAE 格式） | 检查点④：新耦合点确认 |
| 5 | Q 分析（质量属性） | 特性需求 + HTSM 维度 | 质量属性测试点 | — |
| 6 | CAE 聚合 | M+F+Q 测试点 | 逻辑用例（LC） | — |
| 7 | 设计计划 | 逻辑用例 + CAE→PPDCS 推断 | 设计计划（方法匹配 + 优先级） | 检查点③：设计计划确认 |
| 8 | PPDCS 五方法设计 | 设计计划 + 因子库 | 物理用例（16 列标准格式） | — |
| 9 | 双层覆盖率验证 | 物理用例 + 需求基线 | 覆盖率报告（SR→TP→LC→PC） | — |
| 10 | 交付物生成 | 全部设计产物 | 测试方案 + 测试用例 | 检查点⑤：最终交付确认 |

> ptm-tde 的 12 步流程中，人工在 5 个检查点介入确认。Step 2 阶段不追求全自动，关键是每个检查点的输出质量。

#### 4.3.2 ptm-te 任务领取与用例执行

| 步骤 | 活动 | 输入 | 输出 | 人工职责 |
|:---:|------|------|------|---------|
| 1 | 任务领取 | 禅道中的测试任务 | 已分配的用例清单 | 确认任务范围和工作量 |
| 2 | 用例解析 | 用例内容 | 结构化行动计划（配置步骤 + 流量动作 + Check 点） | 确认理解正确 |
| 3 | 环境准备 | 逻辑 Topo 需求 | 映射后的物理 Topo + 设备清单 | 切换组网、确认设备可用 |
| 4 | 配置下发 | 行动计划中的配置步骤 | 防火墙配置完成状态 | 调用原子能力（防火墙配置），确认关键配置 |
| 5 | 流量发送 | 行动计划中的流量动作 | 流量执行结果 | 调用原子能力（仪表打流），确认流量参数 |
| 6 | 结果判定 | 配置回显 + 流量结果 + 日志 | Pass/Fail + 判定依据 | 人工确认关键判定，异常时初步定位 |
| 7 | 执行记录 | 全流程操作记录 | 结构化执行日志（配置/流量/结果/异常上下文） | 补充人工观察和备注 |
| 8 | 工具需求反馈 | 执行中发现的工具缺失 | 工具开发需求（提交 ptm-tae） | 描述需求场景和期望接口 |

> **关键**：ptm-te 在第 7 步产出的结构化执行日志是 ptm-tae 自动化翻译的核心原料——记录越完整，翻译质量越高。

#### 4.3.3 ptm-tae 自动化翻译与工厂运行

| 步骤 | 活动 | 输入 | 输出 | 人工职责 |
|:---:|------|------|------|---------|
| 1 | 执行记录读取 | ptm-te 的结构化执行日志 | 可翻译的步骤序列 | 确认哪些用例适合自动化 |
| 2 | 步骤翻译 | 步骤序列 | 单文件 Python 脚本（`test_case_xxx.py`） | 复查关键逻辑是否正确 |
| 3 | 脚本验证 | 生成的 Python 脚本 | 验证执行结果（与手工结果比对） | 确认首次运行结果一致 |
| 4 | 工厂注册 | 验证通过的脚本 | 自动化工厂用例库条目 | 配置 Mark 标签和用例分组 |
| 5 | 回归触发 | 新版本转测通知 | 自动触发对应脚本执行 | 关注失败通知 |
| 6 | 失败处理 | 失败用例 | 自动分析 + 尝试修复 + 推送通知 | 自动修复失败时人工介入 |
| 7 | 报告汇总 | 全量执行结果 | 回归报告（通过率/失败详情/趋势） | 确认报告发送范围 |

> **自动化翻译策略**：优先翻译执行频率高、步骤明确、无复杂人工判断的用例。Step 2 阶段不追求 100% 自动化覆盖率，跑通最小闭环后再扩展。

  │     └── 反馈 → ptm-tde（用例设计评审）
  ├── 设计阶段 → ptm-tde（12 步流程）
  │     └── 反馈 → ptm-te（用例执行）
  ├── 执行阶段 → ptm-te（调用工具执行）
  │     └── 反馈 → ptm-tae（自动化沉淀）
  └── 质量检查 → ptm-qa（过程审计 + 数据采集 + 趋势报告）

人工职责：在 ptm-tm 制定计划、ptm-tse 输出策略、ptm-tde 交付用例、
         ptm-qa 输出质量报告时介入确认，其余流程由 Agent 自主调度
```

### 4.5 Step 4 — Autopilot 阶段

**目标**：六个 Agent 全部完成开发，形成全自动闭环。从禅道接入任务开始，自动调度子 Agent 完成全流程测试，人工仅检查异常。

**阶段特征**：Agent 全自主运行——人工不再参与流程控制和任务分配，角色从"操作者"和"调度者"彻底转变为"监督者"。只有当 Agent 无法处理的异常出现时才介入。

**核心流程**：

```
禅道项目 / 版本转测通知（自动触发）
           │
           ▼
    ┌─────────────────────────────────────────────────────────┐
    │                    ptm-tm（自动调度）                     │
    │                                                         │
    │  自动制定计划 → 自动调度 sub-agent → 自动汇总报告          │
    └──┬──────────────┬──────────────┬──────────────┬─────────┘
       │              │              │              │
       ▼              ▼              ▼              ▼
   ┌────────┐   ┌────────┐   ┌────────┐   ┌──────────────┐
   │ptm-tse │   │ptm-tde │   │ptm-te  │   │ptm-qa        │
   │需求分析│   │用例设计│   │用例执行│   │质量检查       │
   │+ 策略  │   │(自动)  │   │(自动)  │   │(自动)         │
   └────────┘   └────────┘   └───┬────┘   └──────────────┘
                                 │
                         执行记录自动沉淀
                                 │
                                 ▼
                          ┌──────────────┐
                          │   ptm-tae    │
                          │ 自动化翻译    │
                          │ + 回归运行    │
                          │ + 自动修复    │
                          └──────────────┘
```

**全流程自动化**：

| 阶段 | Agent | 自动化能力 |
|:---:|:---:|------|
| 任务接入 | ptm-tm | 从禅道自动获取项目/版本任务，生成测试计划 |
| 需求分析 | ptm-tse | 自动解析需求文档 → 分解特性 → 制定策略 |
| 用例设计 | ptm-tde | 自动执行 12 步流程 → 输出测试方案 + 用例 → 回写禅道 |
| 用例执行 | ptm-te | 自动领取任务 → 映射 Topo → 调用工具执行 → 记录结果 |
| 自动化沉淀 | ptm-tae | 自动翻译执行记录 → 生成脚本 → 注册工厂 → 回归运行 |
| 质量检查 | ptm-qa | 自动审计 → 采集数据 → 生成质量报告 |
| 异常处理 | 全部 | Agent 自行处理已知异常；未知异常才通知人工 |

**人工职责**：
- 每日查看 ptm-tm 汇总的项目执行摘要
- 处理 Agent 标记的异常（无法自动修复的脚本、未识别的错误模式）
- 审核关键交付物（测试方案、质量报告）
- 持续优化 Agent 的提示词和知识库

## 六、风险与下一步行动

### 6.1 关键风险

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|:---:|---------|
| 工具开发依赖硬件环境 | 开发阻塞 | 高 | 优先用 Mock/模拟环境，验证后再接入真机 |
| ptm-tde 复杂度过高 | 理解延迟 | 中 | 先跑通 M 分析（单功能），再深入 F/Q |
| 角色间 Skill 边界模糊 | 重复开发 | 中 | `team-design/skill-map.md` 中先锁死公共/专属边界 |
| 原子操作 CLI 与数据同仓库 | 版本耦合 | 中 | sparse-checkout 排除 src/ + release tag 分 `cli-v*` 和 `atoms-v*` |
| git 认证配置门槛 | CLI 覆盖率低 | 中 | CLI 认证失败时输出详细 SSH/HTTPS 配置引导 |
| 消费方 Python/uv 环境差异 | CLI 安装失败 | 中 | preflight 前置检查 + 手册提供环境配置步骤 |
| 原子操作 schema 后续扩展 | 已有 YAML 不兼容 | 低 | `schema_version` 字段 + semver + CHANGELOG 记录破坏性变更 |
| Windows 平台适配问题 | 部分工具不可用 | 中 | `pathlib` 统一路径处理 + 平台感知缓存路径 |
